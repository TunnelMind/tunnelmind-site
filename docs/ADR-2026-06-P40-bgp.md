## Context

P40 Phase 8 was written as "a `bgp_events` table + a **bgp-monitor container** +
a `get_bgp_events` MCP tool." The intent is the witnessability pillar's routing
dimension: a public, append-only record of when a watched network resource
(prefix/ASN) shows a hijack-shaped routing event — an unexpected origin AS, an
RPKI-invalid announcement, a withdrawal.

Two pieces of reality shape the implementation:

1. **A monitor "container" is the wrong shape.** A routing poll runs a handful
   of times a day; an always-on VPS Docker service would be idle compute. This
   mirrors the Phase 4.3 drift-correction, where the plan's `threat_intel_cache`
   table became an Augur *source* rather than new infrastructure.
2. **The data-api Worker already has a `scheduled()` cron handler** (the Sigil
   P29 D1→Supabase mirror, daily 03:30 UTC). A second schedule on the same
   Worker, branched by `event.cron`, costs nothing new.

## Decision

- **Storage (migration 027, additive).**
  - `bgp_watchlist` — what we monitor. Service-role only (RLS, no policies):
    exposing the fleet's announcing prefixes would let an adversary enumerate
    sensor infrastructure, the same rationale `oai_sensors` omits IPs (mig 020).
    Each row carries a self-baselined `baseline_origins int[]`.
  - `bgp_events` — observed anomalies. **Public-read** (like `sensor_observations`,
    mig 020): routing anomalies are public observations. Service-role write only.
  - `recent_bgp_events(resource, since_ms, limit)` — a row-cap-immune `jsonb`
    scalar read RPC, `SECURITY DEFINER` + service-role-only, consistent with
    `dark_pool_risk` (023) / `traverse_supply_path` (025).
- **Monitor = a second cron on the existing data-api Worker**, not a container.
  `api/bgp-monitor.js`, cron `17 */6 * * *` (every 6h). For each enabled
  watchlist prefix it asks RIPEstat — RIPE NCC, the non-profit RIR, per
  `feedback_no_big_tech`, the same source as the `/api/corpus/asn` tab —
  `prefix-overview` (origins), `routing-status` (RIS-peer visibility), and
  `rpki-validation` (per origin). Writes to Postgres, never KV
  (`feedback_no_freetier_loadbearing`).
- **Self-baselining.** First sight of a prefix records its visible origin set as
  the baseline and emits nothing. Later polls diff against it:
  `origin_change` (new origin AS; `critical` if that announcement is also
  RPKI-invalid, else `high`), `rpki_invalid` (a baseline origin gone invalid,
  `high`), `withdrawn` (no longer announced, `medium`). `new_more_specific` and
  `visibility_drop` are reserved in the enum for a later monitor pass.
- **Read surface.** `GET /v1/bgp/events?resource=&since_ms=&limit=` on
  data.tunnelmind.ai (`api/routes/bgp.js`); MCP tool `get_bgp_events`
  auto-derived from `openapi.yaml` onto mcp-data (54→55 tools). An empty
  `events[]` is the honest "all clear", not a null.
- **Manual trigger.** `POST /v1/admin/bgp/poll` (ADMIN_KEY / BGP_MONITOR_KEY),
  mirroring `/v1/admin/sigil/sync`, for backfill + post-deploy verification.

## Watchlist policy

Seeded with two stable, well-known prefixes purely as monitoring **targets**
(observation, not infrastructure use — `feedback_no_big_tech` governs what we
build *on*, not what we *watch*). They establish a baseline and report stable,
proving the pipeline end-to-end. The watchlist is meant to grow, in priority
order, with: `reason='fleet'` (our own sensor prefixes — witnessability of our
footprint) and `reason='ssp'/'publisher'` (high-value supply-chain resources
whose hijack is a Sigil integrity event). Curating those from the corpus is a
Phase 8 follow-up.

## Consequences

- New routing dimension on the trust graph with zero new always-on infra.
- RIPEstat is a third-party dependency on a poll path (not a request hot path);
  a failed poll records `last_error` on the watchlist row and emits no event —
  it never fabricates an anomaly or 500s a read.
- The 6h cadence can miss sub-6h transient hijacks; acceptable for v1, tunable
  by editing the cron. A future pass may add RIPE RIS Live streaming if the
  signal justifies the always-on cost.
