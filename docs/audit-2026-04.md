## 2. Current Routes

The app uses a **hash-based router** (`/#/path`). All routing is handled client-side; no server-side routing rules exist.

| Hash Route | Page Component | `t/` Prefix | Description |
|---|---|---|---|
| `/#/` (default) | `Dialog.jsx` | `t/dialog` | Main collaborative writing space with Shadow Graph |
| `/#/products` | `Products.jsx` | `t/products` | Tool catalog: NetProbe, Radar, API, NetShell, Personal tier |
| `/#/roadmap` | `Roadmap.jsx` | `t/roadmap` | Development timeline (published document) |
| `/#/contributors` | `Contributors.jsx` | `t/contributors` | Leaderboard + compensation framework + Stripe Connect |
| `/#/about` | `About.jsx` | `t/about` | Origin story and philosophy |
| `/#/receipt` | `Receipt.jsx` | `t/receipt` | Surveillance Receipt generator (fully client-side) |

**`t/` prefix usage:** The prefix appears in page descriptions, breadcrumbs rendered by `SubredditNav.jsx`, and `PageDesc.jsx` components. It does NOT appear in the actual URL hash routes — it is purely cosmetic display text. Removal requires editing display strings in those components, not the router config.

## 4. Supabase Tables

**Current sync status:** Most tables are defined in migrations but not yet actively synced. Only `contributors` (read) and `contribution_ledger` (write) are wired in Phase 1.

| Table | Purpose | Key Columns | Active Sync |
|---|---|---|---|
| `transmissions` | Document pages | id, page, section_key, title, created_at | No (seeded) |
| `sentences` | Individual sentences | id, transmission_id, sentence_key, content, is_redacted, redaction_threshold | No |
| `sentence_votes` | Sentence up/down votes | id, sentence_id, fingerprint, direction (-1/1) | No |
| `corrections` | Proposed text corrections | id, sentence_id, proposed_text, contributor_fingerprint, is_applied | No |
| `correction_votes` | Votes on corrections | id, correction_id, fingerprint, direction | No |
| `annotations` | Threaded comments | id, sentence_id, parent_id, content, contributor_fingerprint, is_author | No |
| `annotation_votes` | Votes on annotations | id, annotation_id, fingerprint, direction | No |
| `redaction_votes` | Declassification votes | id, sentence_id, fingerprint, created_at | No |
| `contributors` | User profiles + payout | id, handle, fingerprint, email, email_verified, stripe_* fields, identity_tier | **Yes (read)** — `state.jsx` loads Stripe status on auth |
| `contribution_ledger` | Append-only scoring log | id, contributor_id, contributor_fingerprint, action_type, target_id, points, created_at | **Yes (write)** — `state.jsx` inserts when `points > 0` and authenticated |
| `score_weights` | Configurable action weights | action_type (PK), weight, updated_at | No (hardcoded in scoring.js) |
| `payout_ledger` | Monthly credits/debits | id, contributor_id, amount_cents, reason, payout_run_id, created_at | No |
| `payout_runs` | Monthly payout cycles | id, period (PK), status, total_revenue_cents, business_reserve_cents, contributor_pool_cents | No |

**Views:**
- `sentence_net_scores` — sum of votes per sentence
- `correction_net_scores` — sum of votes per correction
- `redaction_progress` — declassification progress per sentence
- `contributor_scores` — materialized, total weighted points per contributor

**Split risk assessment:**

| Risk Item | Severity | Detail |
|---|---|---|
| `localStorage` key collision | Medium | Both sites would share `tm_state` key if opened in same browser. Needs namespacing (`tm_state` vs `alloy_state`). |
| Supabase schema coupling | Medium | All tables in one schema. Alloy needs a separate schema or project. `contribution_ledger` must remain unified for global scoring. |
| Product data hardcoded | Medium | `Products.jsx` has hardcoded `LIVE_TOOLS` / `PERSONAL_TOOLS` arrays. Extract to API or shared package before split. |
| Seeded documents hardcoded | Low | `SEEDED_DOCUMENTS` in `state.jsx` are site-specific. Each site needs its own seeds. |
| Stripe account | Low | Payout account is unified — desired behavior. Both sites can share the same Stripe account via unified Supabase `contribution_ledger`. |
| CORS allowlist | Low | `functions/_middleware.js` will need `alloy.tunnelmind.ai` added. |
| `VITE_AUTHOR_KEY` unused | Trivial | Delete from env and code. |
| Hash router | No risk | Zero server-side dependency — clean split. |
| No Redux or heavy state | No risk | `useReducer` + context is easily duplicated or extracted. |
| No CDN dependencies | No risk | No external CDN scripts in `index.html`. |

## 8. Shadow Graph Separability Assessment

The Shadow Graph is **clearly separable.** All Shadow Graph code is isolated to:

- `src/pages/Dialog.jsx` — the host page
- `src/components/InteractiveSentence.jsx`
- `src/components/AnnotationPanel.jsx`
- `src/components/CorrectionThread.jsx`
- `src/components/RedactionBlock.jsx`
- State slices in `src/lib/state.jsx`: `sentenceVotes`, `corrections`, `annotations`, `redactionVotes`, `appliedTexts`
- Supabase tables: `sentences`, `sentence_votes`, `corrections`, `correction_votes`, `annotations`, `annotation_votes`, `redaction_votes`, `transmissions` (and views)

**None of these are imported by the product pages** (Products, Roadmap, About, Receipt). The `DocumentEditor.jsx` component hosts Shadow Graph via props — when Shadow Graph features are disabled, it renders as a plain document viewer.

The only shared dependency is `src/lib/scoring.js` (needed by both sites for contribution weighting) and `contribution_ledger` (the unified scoring table).

|---|---|

| Tier | Multiplier | Requirement |
|---|---|---|
| `email` | 0.5× | Email verified |

**Revenue waterfall:**
- 15% business reserve
- 85% contributor pool
- Monthly payout cycle, $10 minimum threshold, sub-threshold balances carry forward

## 10. Pre-Split Decisions Required

Before Phase 1 coding begins, the following decisions must be made:

1. **Supabase:** Separate project for Alloy, or separate schema in the same project?
   - Recommendation: Same project, separate schemas (`public` for tunnelmind, `alloy` schema for alloy). Unified auth realm, shared `contribution_ledger`, Stripe account stays unified.

2. **Repo structure for Alloy:** New standalone repo, or second workspace in a monorepo?
   - See Phase 2 prompt for recommendation request.

3. **Shared design token package:** Shared npm package, shared git submodule, or symlinked folder?
   - Recommendation: Shared folder in a monorepo with path alias in Vite configs.

4. **`localStorage` namespacing:** Confirm `tm_state` stays for tunnelmind.ai and Alloy uses `alloy_state`.

5. **Stripe:** Confirm unified Stripe account. Alloy contributors earn in the same pool.

6. **CORS:** Confirm Alloy can call `functions/api/stripe/*` on tunnelmind.ai, or Alloy needs its own Stripe Functions.
